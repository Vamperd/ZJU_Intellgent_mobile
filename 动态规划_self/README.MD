# 动态规划_self 项目说明

## 1. 项目概述

本项目是一个基于 Python 的机器人导航与规划实验程序，运行环境面向机器人足球/导航仿真平台。程序的核心目标不是做复杂任务编排，而是完成一个非常明确的闭环过程：

1. 从视觉模块实时接收场上机器人位置与姿态信息；
2. 以己方机器人 `blue_robot[0]` 作为受控对象；
3. 依据目标点和周围障碍物，实时生成导航方向；
4. 在正常情况下使用人工势场法（APF）进行局部导航与避障；
5. 在机器人疑似陷入局部极小值或“卡住”时，触发 PRM 概率路网规划作为脱困机制；
6. 使用 PID 控制器将期望方向转换为角速度控制命令；
7. 通过 UDP 向仿真平台发送速度指令，并把路径和关键点绘制到调试界面。

从实现思想上看，这个项目属于一种“局部反应式导航 + 卡死时在线重规划”的混合式导航方案。它并不是经典意义上的“动态规划（Dynamic Programming）”算法实现，而是一个面向动态场景的实时导航程序：障碍物位置来自实时视觉输入，导航力和脱困路径会随环境变化在线更新。

## 3. 系统运行流程

程序主循环位于 `main.py`，整体执行过程如下：

### 3.1 初始化阶段

在 `main.py` 中依次创建以下对象：

- `Vision()`：开启视觉接收线程；
- `Action()`：准备向控制端口发送机器人运动命令；
- `Debugger()`：准备向调试端口发送绘图信息；
- `PRM()`：构造脱困路径规划器；
- `PID()`：构造角度控制器。

### 3.2 实时循环阶段

主循环中每次都执行以下步骤：

1. 读取当前机器人位置；
2. 根据状态变量 `flag` 决定当前要去的目标点；
3. 调用 `Force()` 计算当前时刻的合力方向；
4. 根据合力向量得到期望朝向 `w_expect`；
5. 根据机器人最近一小段位移估计当前实际朝向 `w_real`；
6. 将两者差值送入 PID，生成转向控制量；
7. 若机器人接近目标点，则切换到另一目标点继续往返；
8. 若机器人长时间几乎没有移动，则认为可能陷入局部极小，触发 PRM；
9. 将当前速度与角速度通过 `Action.sendCommand()` 发给仿真平台；
10. 将机器人、目标点、规划路径通过 `Debugger` 发送到调试界面。

这形成了一个完整的实时闭环：

```text
视觉输入 -> 状态解析 -> 局部导航/重规划 -> PID 控制 -> 指令输出 -> 调试显示
```

## 4. 各文件代码解析

## 4.1 `vision.py`：视觉感知模块

### 主要功能

`vision.py` 负责从本地 UDP 端口接收视觉系统发来的 protobuf 数据包，并将数据转换为程序内部可直接访问的机器人状态对象。

### 核心实现

#### 1）网络接收

- 使用 `socket.AF_INET + socket.SOCK_DGRAM` 创建 UDP 套接字；
- 绑定地址 `127.0.0.1:23333`；
- 启动后台线程 `receive_vision()` 持续接收数据。

这意味着视觉数据接收不会阻塞主线程，主控制循环可以持续运行。

#### 2）protobuf 解析

接收到的数据会被解析成 `Vision_DetectionFrame`，随后调用 `parse_vision()` 把结构化信息写入：

- `self.blue_robot[0..15]`
- `self.yellow_robot[0..15]`

每个机器人对象 `Robot` 中保存了：

- 是否可见 `visible`
- 滤波后位置 `x, y`
- 滤波后速度 `vel_x, vel_y`
- 姿态角 `orientation`
- 原始视觉量 `raw_x, raw_y, raw_vel_x, raw_vel_y, raw_orientation`

#### 3）己方机器人选择

`my_robot` 属性固定返回 `self.blue_robot[0]`，说明该程序只控制蓝方 0 号机器人。

### 在导航中的作用

这是整个项目的数据入口。导航与规划模块并不维护自己的世界模型，而是直接依赖视觉线程持续更新的机器人位置。因此整个系统具备“基于当前场景实时重算”的动态特性。

---

## 4.2 `action.py`：动作输出模块

### 主要功能

`action.py` 负责把控制结果封装成 protobuf 指令，并通过 UDP 发给仿真平台。

### 核心实现

`sendCommand(vx=0, vy=0, vw=0)` 中会：

1. 创建 `Robots_Command()`；
2. 添加一个 `Robot_Command`；
3. 指定 `robot_id = 0`；
4. 写入：
   - `velocity_x = vx`
   - `velocity_y = vy`
   - `velocity_r = vw`
5. 关闭踢球、吸球等功能；
6. 发往 `localhost:50001`。

### 在导航中的作用

导航模块计算出的“想往哪走”最终必须转换为机器人运动指令。这个文件承担了控制链路的最后一跳。

值得注意的是：虽然协议支持 `vx / vy / vw` 三个速度分量，但主程序实际主要使用的是：

- 前向速度 `vx`
- 角速度 `vw`

侧向速度 `vy` 基本没有被利用，因此当前实现更像“差速/朝向驱动”的导航，而不是充分利用全向机器人的平面速度控制能力。

---

## 4.3 `pid.py`：转向控制模块

### 主要功能

`pid.py` 提供一个标准 PID 控制器类。项目中它没有用于位置闭环，而是用于“方向误差 -> 角速度命令”的控制。

### 核心实现

PID 的标准形式如下：

```text
output = PTerm + Ki * ITerm + Kd * DTerm
```

代码中的默认参数为：

- `Kp = 0.3`
- `Ki = 0`
- `Kd = 0.001`

这说明当前控制器以比例项为主，微分项为辅，积分项关闭。这样做的意图通常是：

- 保持转向响应直接；
- 避免积分累积带来的震荡；
- 用少量微分抑制方向误差的快速变化。

### 在导航中的作用

在 `main.py` 中，程序会先得到：

- 期望朝向 `w_expect`
- 实际朝向 `w_real`

然后计算误差：

```text
err = w_expect - w_real
```

接着送入 PID 得到 `w_pid.output`，最终用于生成旋转速度：

```text
vw = -w_pid.output * 10
```

因此，PID 负责让机器人尽快把朝向调整到“导航向量要求的方向”。

## 4.4 `func.py`：人工势场导航核心

这是整个项目中最关键的导航代码之一。

### 主要函数

- `Force(robot_position, vision, goal)`
- `angle(totalv)`

其中 `Force()` 负责生成导航向量，`angle()` 负责把向量转换为角度。

### 4.4.1 人工势场法的基本思想

人工势场法（Artificial Potential Field, APF）把导航问题转化为“虚拟力场”问题：

- 目标点对机器人产生吸引力；
- 障碍物对机器人产生斥力；
- 两者叠加后得到一个合力向量；
- 机器人沿合力方向运动。

在本项目中：

- 吸引力负责“朝目标前进”；
- 斥力负责“远离周围机器人”；
- 合力决定当前时刻的期望运动方向。

### 4.4.2 吸引力的实现

目标吸引力计算逻辑为：

```python
if d_goal <= d_a:
    Fatt = -2 * k_a * (robot_position - goal)
else:
    Fatt = -2 * k_a * d_a * (robot_position - goal) / d_goal
```

这里体现了两段式设计：

#### 1）距离目标较近时

吸引力与位置误差近似线性相关，机器人会更精细地收敛到目标点。

#### 2）距离目标较远时

吸引力被限制在一个饱和值，不再随着距离无限增大。这样做的目的是避免目标太远时吸引项过大，导致控制失稳或者完全压制避障斥力。

相关参数：

- `k_a = 0.00055`：吸引力增益
- `d_a = 1000`：吸引力饱和距离

### 4.4.3 斥力的实现

斥力针对周围机器人逐个计算。代码中选取的是黄方机器人位置：

```python
Position_Yellow = np.array([[vision.yellow_robot[i].x, vision.yellow_robot[i].y] for i in range(0,16)])
Position = Position_Yellow
```

对于每一个障碍物，先计算其与机器人之间的距离 `d_frep[i]`。如果距离小于障碍影响半径 `d_o`，则施加斥力：

```python
Frep1 = k_b * (1.0/d - 1.0/d_o) * 1.0/d^2 * direction
```

其含义是：

- 障碍离得越近，斥力越大；
- 障碍超出影响半径后，不再产生斥力；
- 斥力方向由障碍指向机器人。

相关参数：

- `k_b = 7e7`：斥力增益
- `d_o = 800`：障碍影响范围

### 4.4.4 合力如何用于导航

`Force()` 最终返回：

```text
F = Fatt + Frep_sum
```

在 `main.py` 中：

```python
totalv = F * 500
```

也就是说，势场合力先被放大，再作为期望速度方向。接下来程序只取它的方向角，并将这个方向交给 PID 控制器进行转向控制。

### 4.4.5 `angle()` 的作用

`angle(totalv)` 根据二维向量计算其夹角，用于把“力/速度向量”转换成“朝向角”。

这一步在导航中非常重要，因为主控制器并不是直接对二维速度做闭环，而是先统一到“应该朝哪个方向转过去”。

## 4.5 `prm.py`：PRM 概率路网脱困规划

这是项目中另一个最关键的文件，负责在人工势场陷入局部极小值时提供一条绕开障碍的临时路径。

## 4.5.1 为什么需要 PRM

人工势场法虽然实时、简单、计算快，但有一个经典问题：

- 可能陷入局部极小值；
- 可能在障碍间反复振荡；
- 可能看似有目标吸引力，但合力方向不足以带机器人脱困。

为了补足这个缺点，项目引入 PRM（Probabilistic Roadmap）作为“失败保护机制”：

- 正常时依然用 APF；
- 卡住时才触发 PRM；
- PRM 找到一条路后，不直接全程跟踪，而是选择路径中的中间点作为临时目标，让 APF 继续接管局部导航。

这就是项目“导航 + 规划 + 动态实现”的核心结合点。

## 4.5.2 `plan()`：规划总入口

`plan()` 做了 4 件事：

1. 从视觉中提取障碍物；
2. 建立障碍 KDTree；
3. 采样自由空间点；
4. 生成路网并用 Dijkstra 搜索路径。

函数最终返回：

- `path_x, path_y`：规划路径
- `road_map`：路网连接关系
- `sample_x, sample_y`：采样点集

### 障碍物来源

PRM 中把以下对象当作障碍：

- 可见的蓝方机器人（排除 `id == 0` 的自己）
- 可见的黄方机器人

这说明 PRM 的障碍建模比 APF 更完整一些。

## 4.5.3 `sampling()`：自由空间采样

PRM 的第一步是在空间中随机撒点，但这些点不能落在障碍附近。代码做法是：

- 随机选取“目标点附近”或“某个障碍物附近”作为采样中心；
- 在中心周围一定范围内随机生成候选点；
- 利用 KDTree 查询该点到最近障碍的距离；
- 如果距离大于 `robot_size + avoid_dist`，则保留为有效采样点。

参数含义：

- `N_SAMPLE = 50`：采样点数量
- `robot_size = 200`：机器人尺寸近似半径
- `avoid_dist = 300`：额外避障安全距离

因此，采样点至少要距离障碍物 `500` 以上才会被保留。

最后，程序还会把：

- 起点 `start`
- 终点 `goal`

追加到采样点集合中，确保图搜索一定包含起点和目标点。

## 4.5.4 `generate_roadmap()`：构建路网

得到采样点后，需要把点连接成图。代码的基本思路是：

1. 建立采样点 KDTree；
2. 对每个采样点寻找附近邻居；
3. 按距离从近到远尝试连接；
4. 只有连线不碰障时，才加入边；
5. 每个点最多保留 `KNN = 10` 个邻居。

这一步构建出的 `road_map` 本质上就是一个稀疏图，后续 Dijkstra 在这个图上找路。

## 4.5.5 `check_obs()`：边碰撞检测

点能否连接，取决于两点连线是否安全。`check_obs()` 的思路是沿连线方向离散采样：

1. 计算边长与方向；
2. 若边长超过 `MAX_EDGE_LEN`，直接判为不可连；
3. 按步长 `robot_size + avoid_dist` 沿边前进；
4. 每走一步就检查当前位置到最近障碍的距离；
5. 若距离小于安全阈值，则说明碰撞，边无效；
6. 终点也会再检查一次。

这相当于做了一个简单但有效的线段碰撞检测。

## 4.5.6 `dijkstra_search()`：图搜索

路网建好后，程序采用 Dijkstra 算法求从起点到终点的最短路径。

实现过程如下：

- 将起点加入 `openset`；
- 每次取出当前总代价最小的节点；
- 扩展其邻接边；
- 若找到更优路径则更新；
- 到达终点后停止；
- 根据父节点回溯构造路径。

这里的边代价采用欧氏距离，因此得到的是一条基于路网的几何最短路径。

## 4.5.7 `search()`：备用搜索逻辑

当 `dijkstra_search()` 返回的路径长度为 0 或 2 时，代码会尝试调用 `search()`。从设计意图看，它希望：

- 当没有理想路径时，至少找一个较合适的中间采样点；
- 或者当路径过于简单时，强行构造一个脱困中继点。

不过，从代码实现上看，这个函数存在明显问题，鲁棒性较弱，因此它更像一个实验性备用方案，而不是成熟的第二规划器。

## 5. `main.py` 详解：主控逻辑如何把各模块串起来

`main.py` 是项目最核心的调度文件。下面按逻辑顺序解释其关键内容。

## 5.1 状态变量含义

主程序中有几个重要变量：

- `goal`：当前目标点
- `flag`：导航状态标志
- `count`：启动阶段保护标志
- `w_real`：估计的实际朝向
- `recode`：记录上一个参考位置
- `repeat`：连续卡住计数

其中 `flag` 很关键：

- `flag = 1`：朝 `[-2400, -1500]` 导航
- `flag = -1`：朝 `[2400, 1500]` 导航
- `flag = 0`：处于 PRM 脱困状态

也就是说，这个变量同时承担了“目标切换状态”和“是否正在脱困”的双重职责。

## 5.2 正常导航：人工势场驱动

每次循环中，首先根据 `flag` 选择当前目标：

```python
if flag == 1:
    goal = np.array([-2400, -1500])
elif flag == -1:
    goal = np.array([2400, 1500])
```

然后计算势场合力：

```python
F = Force([robot.x, robot.y], vision, goal)
```

再把合力放大为期望速度向量：

```python
totalv = F * 500
```

这一步说明：程序没有显式构造一条连续轨迹，而是每个周期重新根据场景计算一股“应该往哪走的力”。这正是反应式导航的典型特征。

## 5.3 朝向估计与角度误差

程序没有直接使用视觉返回的机器人 `orientation` 作为当前朝向，而是采用最近位移估计法：

```python
v_real = [robot.x - robot_last_x, robot.y - robot_last_y]
```

若 x 方向位移不为 0，就调用 `angle(v_real)` 得到 `w_real`。

与此同时，势场合力的角度被视为期望朝向：

```python
w_expect = angle(totalv)
```

两者差值：

```python
err = w_expect - w_real
```

之后还进行了 `[-pi, pi]` 范围内的误差归一化，避免跨越角度边界时出现大跳变。

## 5.4 PID 如何作用于速度控制

误差进入 PID 后得到 `w_pid.output`。主程序根据这个输出同时调整：

- 角速度
- 前进速度

具体控制规律为：

```python
vw = -w_pid.output * 10
vx = ||totalv|| * 2 / (1 + 2 * abs(w_pid.output))
```

这条公式很有代表性，说明作者采用了“转得越急，走得越慢”的策略：

- 当方向误差大时，先优先转向，避免一边大偏角一边高速前冲；
- 当方向误差小时，再提高线速度，沿目标方向前进。

这增强了系统的稳定性，也让 APF 的方向输出更容易被机器人执行出来。

## 5.5 到点切换机制

主程序并不是单次导航，而是在两个固定目标之间往返：

```python
elif d_goal < 100:
    flag = -1 * flag
    action.sendCommand(vx=0, vw=10)
    time.sleep(0.2)
```

当机器人距离目标小于 100 时：

- `flag` 翻转；
- 目标点切换到对角位置；
- 同时发送一个短暂旋转命令。

因此程序行为是持续巡航，而不是到达后停止。

## 6. 导航、规划、动态实现的重点说明

这一部分是本项目最值得重点理解的内容。

## 6.1 导航是如何做到的

本项目的基础导航不是“先离线规划一条完整路径，再逐点跟踪”，而是：

- 每一帧都根据当前环境重新计算势场；
- 用目标点产生吸引力；
- 用周围机器人产生斥力；
- 将合力方向视为当前期望运动方向；
- 用 PID 让机器人朝这个方向转过去。

换句话说，它的导航本质是“在线方向引导”。

这种方法的优点是：

- 简单；
- 实时性高；
- 易于应对障碍物的瞬时变化；
- 不需要维护复杂地图。

缺点是：

- 容易陷入局部极小值；
- 很依赖参数调节；
- 对复杂障碍布局不够鲁棒。

## 6.2 规划是如何做到的

规划功能主要体现在 `prm.py` 中的 PRM + Dijkstra。

它不是每一帧都运行，而是在出现“卡住”现象时作为应急机制触发。

PRM 的工作流程是：

1. 根据当前视觉信息收集障碍物；
2. 在自由空间随机采样若干节点；
3. 将可安全连接的节点连成图；
4. 在图上使用 Dijkstra 求一条从起点到目标的可行路径；
5. 从这条路径中取一个中间点，作为新的临时目标点；
6. 让 APF 重新围绕这个中间点继续导航。

这说明本项目中的“规划”并非持续型全局规划，而是“局部导航失败时的临时图搜索规划”。

## 6.3 避障是如何做到的

避障分为两层：

### 第一层：人工势场斥力避障

由 `Force()` 中的斥力项完成。只要障碍物进入作用范围 `d_o = 800`，就会产生把机器人推开的虚拟力。

这是实时、连续、反应式的避障方法。

### 第二层：PRM 碰撞检测避障

在 PRM 脱困阶段，障碍物不仅影响局部力场，还会影响图搜索：

- 采样点若离障碍太近，则被丢弃；
- 两采样点之间若连线穿过障碍安全区，则不允许连边。

因此 PRM 给出的路径是几何上避开障碍物的。

## 6.4 动态实现是如何做到的

如果只看文件名，容易误认为这里的“动态”是动态规划算法；但从代码实现来看，项目中的“动态”主要体现在“实时环境变化下的在线更新”。

具体体现在以下几个方面：

### 1）障碍物是实时变化的

障碍物位置不是写死在地图里，而是来自 `vision.py` 持续接收的视觉数据。也就是说，每一轮导航计算面对的都是当前场上最新位置。

### 2）导航向量是每周期重算的

程序并不会长时间坚持一条旧路径，而是每次循环都重新计算：

- 当前目标位置
- 当前机器人位置
- 当前障碍物分布
- 当前势场合力

这使得机器人可以对环境变化做即时响应。

### 3）卡住后会在线重规划

程序通过检测“当前位置与记录点之间的位移是否长期过小”来判断机器人是否卡住：

```python
d_move < 10
repeat > 200
```

若成立，则调用：

```python
planner.plan(...)
```

并把 PRM 结果中的一个路径节点设为新的临时目标。这是一个典型的“动态重规划”过程：不是预先算好全部路径，而是在运行过程中根据实际效果决定是否重新规划。

### 4）脱困状态不是永久的

程序会计算一个阈值 `d_prm`，当机器人在脱困阶段已经足够接近 PRM 给出的中间目标时，就退出 PRM 状态，恢复原先的总目标导航逻辑。

因此，PRM 不是取代 APF，而是动态插入、动态退出的辅助模块。

---

## 7. 主程序中的动态脱困机制详解

这是本项目实现中非常有特色的一部分。

## 7.1 如何判断“卡住”

程序在每个循环中记录一个参考位置 `recode`，随后比较当前位置与该位置的距离：

```python
d_move = np.linalg.norm(np.array([vision.my_robot.x, vision.my_robot.y]) - recode)
```

若：

- 单次移动距离小于 `10`
- 并且这种情况持续超过 `200` 次

则认为机器人长时间没有有效位移，即有较大概率陷入局部极小值、被障碍困住或控制无法推进。

## 7.2 卡住后如何处理

一旦满足卡住条件，主程序执行：

1. 调用 `planner.plan()` 做 PRM 规划；
2. 获得路径 `path_x, path_y`；
3. 取倒数第二个点作为新的临时目标：

```python
goal = np.array([path_x[l-2], path_y[l-2]])
```

这样做的逻辑是：

- 路径最后一个点通常是当前目标；
- 倒数第二个点是通向目标前的一个过渡点；
- 先引导机器人去这个中继点，更容易绕开当前局部困境。

## 7.3 为什么不直接跟踪整条路径

代码没有做完整路径跟踪器，而是只取一个中间目标点，再交回给 APF。这样做有几个原因：

- 实现更简单；
- 与现有 APF 框架融合容易；
- 只需要让 PRM 提供“脱困方向”，不必完全替代导航主逻辑。

也就是说，PRM 在这里不是主驾驶员，而是“给 APF 提供一个更好目标点的导航顾问”。

## 7.4 如何从 PRM 状态恢复

程序会根据进入脱困状态时与临时目标的距离计算 `d_prm`，作为退出条件。当在 PRM 状态下到达足够接近该中间目标的位置后，`flag` 会恢复为原来的 `_flag`，系统重新回到正常的总目标导航。

这相当于一个很轻量的状态机：

- 正常导航状态
- 脱困规划状态
- 脱困完成后再切回正常状态

## 9. 关键参数总结

## 9.1 势场参数（`func.py`）

- `k_a = 0.00055`：吸引力增益
- `k_b = 7e7`：斥力增益
- `d_o = 800`：障碍影响半径
- `d_a = 1000`：吸引力饱和距离

### 含义

- `k_a` 越大，机器人越倾向于快速靠近目标；
- `k_b` 越大，机器人避障动作越强；
- `d_o` 越大，机器人越早开始规避障碍；
- `d_a` 用于限制远距离时的吸引力大小。

## 9.2 PRM 参数（`prm.py`）

- `N_SAMPLE = 50`：采样点数
- `KNN = 10`：每个节点最多连接 10 个邻居
- `MAX_EDGE_LEN = 5000`：边长上限
- `robot_size = 200`：机器人安全尺寸近似
- `avoid_dist = 300`：额外安全缓冲

### 含义

- 采样点越多，图越丰富，但计算量越大；
- `KNN` 越大，图连通性越强，但碰撞检测也更多；
- `robot_size + avoid_dist` 实际决定 PRM 的安全通行半径。

## 9.3 控制参数（`pid.py` + `main.py`）

- `Kp = 0.3`
- `Ki = 0`
- `Kd = 0.001`
- `vw = -pid_output * 10`
- `vx = ||totalv|| * 2 / (1 + 2 * abs(pid_output))`

### 含义

- 比例项主导朝向纠偏；
- 角误差大时线速度自动减小；
- 实现“先摆正方向，再快速前进”的控制风格。

## 9.4 状态机阈值（`main.py`）

- `d_goal < 100`：判定到达目标
- `d_move < 10`：判定一轮内几乎未移动
- `repeat > 200`：判定已长期卡住，触发 PRM

## 11. 当前代码中的局限与潜在问题

从工程完整性角度看，这份代码更偏实验性质，存在一些值得注意的地方。

## 11.1 导航层面的局限

### 1）人工势场容易陷入局部极小值

这也是项目必须引入 PRM 的原因之一。只靠 APF 很难应对复杂障碍布局。

### 2）没有做轨迹平滑与速度约束

当前控制更像“方向驱动”，而不是严格意义上的轨迹跟踪控制。

### 3）没有进行障碍物运动预测

所谓“动态”更多是实时响应，而不是基于时间维度的动态避障预测。程序只看当前障碍位置，并不预测其未来位置。

## 11.2 实现上的问题

### 1）`vision.py` 中线程启动顺序存在风险

接收线程在某些成员完全初始化前就被启动，理论上有竞态风险。

### 2）主程序没有显式等待有效视觉帧

如果程序启动时视觉数据尚未到达，默认坐标可能仍是 `-999999`，这会影响最初几轮控制计算。

### 3）APF 与 PRM 的障碍建模不完全一致

- APF 主要使用黄方机器人作为障碍；
- PRM 同时考虑黄方与可见蓝方机器人。

两种导航机制对环境的理解并不完全统一。

### 4）`angle()` 没有使用 `atan2`

当前是手工按象限处理，容易在边界条件或零向量情况下产生不够稳健的结果。

### 5）实际朝向估计较粗糙

程序没有直接使用视觉提供的 `orientation`，而是用位移方向近似朝向，这在速度低、停顿、横向移动时可能不够准确。

### 6）`search()` 函数鲁棒性不足

作为 PRM 的备用路径逻辑，它存在明显实现问题，说明该部分尚处于实验状态。

## 12. 项目本质总结

如果用一句话概括，这个项目做的是：

> 在机器人仿真平台中，基于实时视觉输入，使用人工势场进行在线导航与局部避障，并在检测到卡死时调用 PRM 做临时重规划，再通过 PID 输出控制命令驱动机器人运动。

再具体一点，它的核心技术链条是：

```text
实时视觉感知
    -> 势场法生成局部导航方向
    -> PID 调整朝向与速度
    -> 卡死检测
    -> PRM + Dijkstra 生成脱困中继点
    -> 回到势场导航继续执行
```

因此，项目真正的亮点不在于单独实现了某一个复杂算法，而在于把以下几种机制组合成了一个可运行的闭环系统：

- 实时感知
- 局部导航
- 动态避障
- 在线脱困规划
- 闭环控制
- 调试可视化

这也是理解本项目最重要的角度。

本项目是一份典型的导航规划实验代码：

1. 统一 APF 与 PRM 的障碍建模；

3. 引入路径跟踪器而不是只跟踪中间点；
4. 增加速度限幅、加速度约束与轨迹平滑；
5. 改进卡死检测与动态障碍预测机制。

这样可以把当前实验型实现逐步提升为更稳定、更鲁棒的导航系统。
